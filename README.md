# 釣魚預測網站 - 第一步：資料管道 (GitHub Actions)

本階段完成內容：
- `scripts/spots_config.py`：釣點 + 魚種 + 潮汐站對應的共用設定字典
- `scripts/monthly_tide_fetch.py`：工作流 A，每月撈潮汐+月相，算大小潮背景分
- `scripts/frequent_weather_fetch.py`：工作流 B，每3小時撈即時天氣+風向，融合成最終指數
- `.github/workflows/monthly_tide.yml`：工作流 A 的排程設定
- `.github/workflows/frequent_weather.yml`：工作流 B 的排程設定

## 與原規格書的差異說明

寫程式過程中查證天文台官方 API 文件（v1.13, 2025年9月版）後，
發現幾個跟規格書最初假設不同的地方，這裡列出來，避免你之後誤會：

### 1. 沒有結構化的「逐3小時風速/風向」API
規格書工作流B想要「抓取當天最新的即時氣象 JSON（風速、風向、降雨、氣壓）」，
但天文台公開API裡：
- 即時觀測 (`rhrread`) 有溫度、濕度、降雨，**沒有風速風向欄位**
- 唯一有風向資訊的是「9天預報」(`fnd`) 裡的 `forecastWind`，但這是
  **文字描述**（例如「東北風3至4級」），且是**逐日**更新，不是逐3小時。

目前 `frequent_weather_fetch.py` 的做法是解析這段中文文字，抓出方位詞
轉成角度、抓出「X級」轉成大約風速。這是現有免費資料源下的最佳折衷，
已在程式碼註解與輸出JSON的 `data_source_note` 欄位裡誠實標註，
避免之後誤以為這是即時風速觀測值。

之後如果想要更精準的逐小時風速，可考慮的方向（不在本次範圍內）：
- 改抓天文台「天氣現況報告」逐百字文字描述裡的風向關鍵字（一樣是文字解析）
- 串接民航處機場風速API（資料更新頻率較高，但地點侷限在機場附近）
- 改用第三方氣象API（會打破「完全免費、零成本」的限制條件，需另外評估）

### 2. 潮汐站代碼與釣點地理位置不完全對應
天文台只有14個官方潮汐預測站，**沒有「吐露港」「尖沙咀」專屬站**。
目前設定檔裡的對應方式：

| 釣點 | 借用站碼 | 說明 |
|---|---|---|
| 吐露港 | TPK (大埔滘) | 同屬吐露港水系，最接近的官方站 |
| 尖沙咀 | QUB (鰓魚涌) | 同屬維港水域，潮汐特性相近 |
| 長洲 | CCH (長洲) | 長洲本身就有官方站，資料最準確 |

借用站的潮高數字會跟實際釣點有落差（尤其吐露港是內灣，可能有潮時延遲），
這是用「免費官方資料」換取「完全準確」的取捨，之後若要更準，
可以考慮自行做潮時偏移校正（需要實測數據比對，不在本次範圍）。

### 3. 大小潮背景分公式是簡化版經驗公式
工作流A用「當天潮高振幅 (最高潮-最低潮)」線性映射成 0-100 分，
振幅 ≥2.0m 給接近100分（大潮），≤0.5m 給接近0分（小潮）。
這是初版的簡化模型，方便先把資料管道打通，之後可以隨時調整這個公式
（程式碼集中在 `monthly_tide_fetch.py` 的 `calculate_tide_range_score()` 函式），
不需要動到資料抓取的部分。

風向地形加權公式（`frequent_weather_fetch.py` 的 `calculate_wind_terrain_score()`）
同樣是簡化版經驗假設：風直吹向岸 (頂風) 給高分、風吹離岸給中等分。
這兩個公式都標註了清楚的註解說明假設依據，方便你之後依實際釣況反饋調整權重。

## 如何測試（在 GitHub Actions 上）

1. 把這個資料夾推上你的 GitHub repo
2. 到 repo 的 Settings → Actions → General，確認 "Workflow permissions"
   設成 "Read and write permissions"（否則即使 YAML 裡寫了
   `permissions: contents: write`，組織層級設定仍可能擋下推送）
3. 到 Actions 頁籤，手動觸發「工作流 A」(workflow_dispatch按鈕)，
   確認能成功產出 `data/base_tide_*.json`（注意上方「已知問題」一節，
   若天文台API仍異常，這步會失敗，這是預期中的）
4. 工作流A成功後，手動觸發「工作流 B」，確認能產出 `data/tide_score_*.json`
